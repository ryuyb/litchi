# 自动化开发 Agent 需求文档

## 1. 概述

### 1.1 项目目标

构建一个自动化开发 Agent 系统，实现从 GitHub Issue 到 Pull Request 的全流程自动化开发。

### 1.2 核心理念

- **人机协作**：Agent 负责执行，人类负责决策和审核
- **可中断恢复**：每个阶段都支持中断后从断点继续
- **透明可控**：用户可随时介入，Agent 可回退到任意阶段

### 1.3 用户角色定义

| 角色 | 说明 | 职责 |
|------|------|------|
| **Issue 作者** | 创建 Issue 的用户 | 创建 Issue、描述需求、触发流程（@bot）、回答澄清问题 |
| **仓库管理员** | 拥有仓库管理权限的用户 | 回答澄清问题（可替代 Issue 作者）、确认设计方案、审批危险操作、合并 PR |

**角色交互规则**：

```
┌─────────────────────────────────────────────────────────────────────┐
│                        角色交互流程                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Issue 作者                                                          │
│      │                                                              │
│      │ 1. 创建 Issue，描述需求                                       │
│      │ 2. 在 Comment 中 @bot 触发                                   │
│      │ 3. 回答澄清问题（优先）                                       │
│      │                                                              │
│      ▼                                                              │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    Agent 开始工作                            │   │
│  └─────────────────────────────────────────────────────────────┘   │
│      │                                                              │
│      │ 澄清阶段：Issue 作者或管理员均可回答                         │
│      │ 后续阶段（设计确认、审批、合并）：仅管理员                   │
│      │                                                              │
│      ▼                                                              │
│  仓库管理员                                                          │
│      │                                                              │
│      │ 4. 回答澄清问题（若 Issue 作者未响应或需补充）               │
│      │ 5. 确认/拒绝设计方案                                         │
│      │ 6. 审批危险操作                                              │
│      │ 7. 合并 PR                                                   │
│      │                                                              │
│      ▼                                                              │
│  ┌─────────────────────────────────────────────────────────────┐   │
│  │                    WorkSession 完成                          │   │
│  └─────────────────────────────────────────────────────────────┘   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

**设计理由**：
- Issue 作者最了解需求背景，优先回答澄清问题效率更高
- 澄清问题仅涉及需求描述，不涉及代码变更，风险可控
- 设计确认、危险操作审批、PR 合等涉及代码变更的决策仍由仓库管理员控制
- 若 Issue 作者未响应或回答不足，管理员可补充或替代回答

## 3. 详细功能需求

### 3.1 触发机制

| 触发方式 | 说明 | 触发者 |
|---------|------|--------|
| `@bot` 在 Issue Comment 中提及 | 触发 Agent 开始工作 | Issue 作者或仓库管理员 |
| 服务重启后自动恢复 | 检测未完成任务，自动从断点继续 | 系统 |
| 指令 `@bot 继续` | 手动恢复中断的任务 | 仓库管理员 |

**权限检查**：Agent 收到触发请求后，会检查触发者身份：
- 若触发者为 Issue 作者：允许触发，澄清阶段优先由 Issue 作者回答
- 若触发者为仓库管理员：允许触发，后续所有交互由管理员负责

### 3.2 需求澄清阶段

**输入**：GitHub Issue 内容

**输出**：澄清问题 / 需求确认 / 清晰度评分

**行为**：
1. Agent 分析 Issue，识别模糊点、缺失信息
2. 在 Issue Comment 中提问
3. **Issue 作者或仓库管理员**回复后继续分析
   - Issue 作者优先回答（更了解需求背景）
   - 管理员可补充回答或替代回答
4. 每轮对话后计算**需求清晰度评分**
5. 循环直到需求清晰或管理员指令开始设计

**结束条件**：
| 条件 | 处理 |
|------|------|
| 清晰度评分 ≥ 80 | 自动进入设计阶段 |
| 清晰度评分 60-79 | 自动进入设计阶段，但设计方案需仓库管理员确认 |
| 清晰度评分 40-59 | Agent 询问管理员是否开始设计，等待确认 |
| 清晰度评分 < 40 | 必须继续澄清，不能进入设计阶段 |
| 管理员指令"开始设计" | 跳过评分检查，直接进入设计阶段 |

**可恢复信息**：
- 已澄清的问题列表
- 已确认的需求点
- 待澄清问题
- 清晰度评分详情（各维度得分）

### 3.3 设计方案阶段

**输入**：澄清后的需求

**输出**：设计方案文档

**行为**：
1. 分析相关代码模块
2. 输出设计方案（架构、接口、数据流等）
3. 评估复杂度
4. 支持多版本迭代，每次修改生成新版本

**版本管理**：
- 设计方案支持多版本迭代
- 每次用户反馈修改、执行失败回退重新设计，都生成新版本
- 保留历史版本，支持追溯和对比

**复杂度评分标准**：
| 因素 | 权重 | 说明 |
|------|------|------|
| 预估代码量 | 30% | <100行 / 100-500行 / >500行 |
| 涉及模块数 | 25% | 单模块 / 2-3个模块 / >3个模块 |
| 破坏性变更 | 25% | 无 / 接口兼容 / 破坏性变更 |
| 测试覆盖难度 | 20% | 简单 / 中等 / 复杂 |

**确认策略**：
```
if (复杂度评分 > 配置阈值) or (配置强制确认):
    等待人工确认后继续
else:
    自动进入下一阶段
```

**可恢复信息**：
- 已分析的模块
- 当前方案版本
- 设计决策
- 版本变更历史

### 3.4 拆解 Task 阶段

**输入**：设计方案

**输出**：Task List

**行为**：
1. 将设计方案拆解为具体 Task
2. 每个 Task 应可独立实现、独立测试
3. 标注 Task 之间的依赖关系

**可恢复信息**：
- 已拆解的 Task List
- 待拆解部分
- Task 依赖关系图

### 3.5 执行 Task 阶段

**输入**：Task List

**输出**：代码变更

**行为**：
```
for each task in Task List:
    1. 实现该 Task
    2. 执行代码质量验证（根据仓库配置）
       a. 代码格式化（如启用）
       b. Lint 检查（如启用）
       c. 运行测试（如启用）
    3. if 验证通过:
           勾选该 Task
           继续下一个
       else:
           根据失败策略处理：
           - auto_fix: 自动修复重试
           - fail_fast: 通知用户，暂停等待指令
           - warn_continue: 记录警告，继续
           - skip: 跳过验证
           if 仍然失败且策略为 fail_fast:
               通知用户：原因 + 解决方案
               暂停，等待用户指令
```

**执行验证配置**：

| 配置项 | 说明 | 默认值 |
|-------|------|--------|
| `formatting.enabled` | 是否启用格式化 | true |
| `formatting.failureStrategy` | 格式化失败策略 | auto_fix |
| `linting.enabled` | 是否启用 Lint | true |
| `linting.autoFix` | 是否自动修复 Lint 问题 | true |
| `linting.failureStrategy` | Lint 失败策略 | auto_fix |
| `testing.enabled` | 是否启用测试 | true |
| `testing.noTestsStrategy` | 无测试文件时的处理 | warn |
| `testing.failureStrategy` | 测试失败策略 | fail_fast |
| `autoDetection.enabled` | 是否启用自动检测 | true |
| `autoDetection.mode` | 检测模式 | auto_full |

**配置来源优先级**：
1. 仓库级配置（UI 可配置）
2. 自动检测结果
3. 全局默认配置

详细设计见 `execution-validation-design.md`。

**仓库管理员指令**：
- `继续`：从当前 Task 继续执行
- `跳过`：跳过当前 Task，执行下一个
- `修改设计`：回退到设计阶段，重新设计后生成新版本方案
- `回退`：回退到之前阶段重新设计
- `终止`：结束当前任务

**可恢复信息**：
- 已完成的 Task 列表
- 当前执行的 Task
- 失败 Task 及原因
- 代码变更记录

### 3.6 创建 PR 阶段

**行为**：
1. 创建 Pull Request
2. PR 标题：`[Issue-{id}] {需求标题}`
3. PR 描述包含：
   - 变更摘要
   - 设计方案链接
   - Task 完成清单
   - 测试结果
4. **必须人工确认后才能合并**

### 3.7 仓库管理

**功能**：
- 添加/编辑/删除仓库
- 启用/禁用仓库
- 配置仓库级参数覆盖

**权限**：
- 仅仓库管理员可管理仓库配置

**配置项**：
| 配置项 | 说明 | 继承规则 |
|-------|------|---------|
| maxConcurrency | 最大并发数 | 未配置则使用全局配置 |
| complexityThreshold | 复杂度阈值 | 未配置则使用全局配置 |
| forceDesignConfirm | 强制设计确认 | 未配置则使用全局配置 |
| defaultModel | 默认模型 | 未配置则使用全局配置 |
| taskRetryLimit | 任务重试次数限制 | 未配置则使用全局配置 |

**Webhook 路由逻辑**：
1. Webhook 接收时从 payload 提取 `repository.full_name`
2. 查询数据库检查仓库是否已配置且启用
3. 如果不存在或未启用，返回 404 忽略该 Webhook
4. 如果存在且启用，使用仓库级配置（覆盖全局配置）处理请求

**UI 功能**：
- 仓库列表页面：展示所有已配置的仓库及其状态
- 仓库配置页面：添加/编辑仓库配置
- 支持批量操作（批量启用/禁用）

## 4. 中断与恢复机制

### 4.1 中断场景

| 中断原因 | 处理方式 |
|---------|---------|
| 管理员主动暂停 | 保存状态，等待恢复指令 |
| Task 执行失败 | 通知管理员，等待指令 |
| 服务重启 | 自动恢复所有进行中的任务 |
| 外部错误（API 限流等） | 重试 / 通知管理员 |

### 4.1.1 详细失败场景处理

| 场景 | 检测方式 | 自动处理 | 重试策略 | 通知机制 |
|------|---------|---------|---------|---------|
| Agent 调用超时 | 命令执行超时检测 | 增加超时重试1次 | 最多2次 | 超时后 Comment 通知 |
| Agent 进程崩溃 | 进程状态检测 | 尝试恢复会话 | 最多1次 | 立即 Comment 通知 |
| GitHub API 限流 | RateLimit API 检测 | 等待到重置时间 | 等待后继续 | WebSocket 推送等待状态 |
| 测试环境不可用 | 环境预检 | 跳过测试或等待恢复 | 等待5分钟后重检 | Comment 说明跳过原因 |
| 网络连接失败 | 连接错误检测 | 指数退避重试 | 最多3次 | 重试耗尽后通知 |
| 并发资源不足 | 队列状态检测 | 加入等待队列 | 排队等待 | Comment 通知排队状态 |
| 预算超限 | 预算监控 | 使用备用模型 | 不重试 | Comment 通知预算情况 |
| Git 操作失败 | 命令执行错误 | 根据错误类型处理 | 最多2次 | 重试耗尽后通知 |

### 4.1.2 超时配置

| 操作 | 默认超时 | 最大超时 | 超时后行为 |
|------|---------|---------|-----------|
| 澄清阶段 Agent 调用 | 5分钟 | 15分钟 | 暂停等待重新触发 |
| 设计阶段代码分析 | 10分钟 | 30分钟 | 使用缓存分析结果 |
| 设计阶段设计生成 | 15分钟 | 45分钟 | 暂停等待重新触发 |
| 任务拆解 | 10分钟 | 20分钟 | 暂停等待重新触发 |
| 任务执行 | 30分钟 | 60分钟 | 标记失败，自动重试 |
| 测试运行 | 10分钟 | 30分钟 | 视为测试失败 |
| PR 创建 | 5分钟 | 10分钟 | 重试创建 |
| 审批等待 | 24小时 | 48小时 | 自动取消审批请求 |
| 会话总时长 | 72小时 | 168小时 | 强制暂停 |

### 4.2 恢复机制

**手动恢复**：
- 用户在 Issue Comment 中 `@bot 继续`

**自动恢复**：
- 服务启动时检测未完成任务
- 从 `context.json` 读取断点信息
- 自动继续执行

### 4.3 回退机制

```
执行阶段 → 设计阶段 → 澄清阶段
    ↑           ↑          ↑
    │           │          │
    └───────────┴──────────┘
         支持逐级回退
```

**回退规则**：
- 只能逐级回退（执行 → 设计 → 澄清）
- 回退时保留已确认的上下文
- 回退到设计阶段时，重新设计生成新版本（v{n+1}）
- 代码分支标记为废弃，不删除
- 需要用户手动确认后执行回退

### 4.4 幂等性保证

**Webhook 去重**：
- GitHub Webhook 可能因网络问题重发
- 使用 X-GitHub-Delivery 头进行去重
- 已处理的 delivery_id 保存 24 小时
- 重复请求直接返回 200 忽略

**重复触发处理**：
| 场景 | 策略 |
|------|------|
| 同一 Issue 重复 @bot | 若已有活跃 Session，忽略并通知 |
| 同一 Session 重复指令 | 检查当前状态，幂等处理 |
| 重复审批请求 | 检查审批状态，已完成则忽略 |

**幂等性实现要点**：
- 数据库层面：唯一索引保证 delivery_id 不重复
- 业务层面：检查当前状态，避免重复操作
- 返回策略：已处理请求返回 200，避免 GitHub 重试

## 5. 数据持久化策略

### 5.1 Issue Comment

**更新策略**：只在关键节点更新，不实时同步进度

**更新时机**：
- 阶段切换时（澄清完成 → 设计完成 → 开始执行 → 创建 PR）
- 失败/暂停时
- 管理员主动查询进度时（`@bot 进度`）

**存储内容**：
- 当前阶段状态
- 设计方案版本及链接
- 详细进度文件链接
- 最后更新时间

**示例**：
```markdown
## 状态
当前阶段：执行 Task
设计方案：v2

查看详细进度：[tasks.md](链接)

---
最后更新：2026-04-03 12:30
```

**优势**：
- 减少 GitHub API 调用
- 避免 Agent 崩溃导致状态不一致
- 减少不必要的通知打扰

### 5.2 仓库文件

**目录结构**：
```
.litchi/
└── issues/
    └── {issue-id}/
        ├── designs/
        │   ├── v1.md        # 初始方案
        │   ├── v2.md        # 用户反馈后修改
        │   └── v3.md        # 执行失败后重新设计
        ├── tasks.md         # Task 详细内容
        └── context.json     # Agent 执行上下文（进度、输入输出）
```

**设计方案版本命名规则**：
- 文件名格式：`v{version}.md`
- 版本号从 1 开始递增
- 每次修改生成新版本，不覆盖旧版本

**context.json 结构**（Agent 执行上下文）：
```json
{
  "issueId": 123,
  "sessionId": "uuid",
  "currentStage": "implement",
  "agentHistory": [
    {"role": "assistant", "content": "...", "stage": "clarify"},
    {"role": "user", "content": "...", "stage": "clarify"}
  ],
  "clarification": {
    "confirmed": ["需求点1", "需求点2"],
    "pending": []
  },
  "design": {
    "currentVersion": 3,
    "completed": true,
    "complexityScore": 65,
    "requireConfirmation": false,
    "history": [
      {"version": 1, "reason": "初始设计", "createdAt": "2026-04-03T10:00:00Z"},

## 6. 并发控制

### 6.1 并发策略

- 多个 Issue 可同时处理
- 每个 Issue 独立的 Git Worktree
- 最大并发数可配置

### 6.2 并发示意

```
Issue #1 @bot  ──┐
Issue #2 @bot  ──┼──→ 任务队列 ──→ [Worker 1] ──→ Agent 实例 A
Issue #3 @bot  ──┤                    │
Issue #4 @bot  ──┘                    ├──→ [Worker 2] ──→ Agent 实例 B
                                      │
                                      └──→ [Worker 3] ──→ Agent 实例 C
                                              ↑
                                      最大并发数 = 3
```

## 7. Agent 能力要求

### 7.1 核心能力

Agent 需具备以下能力：
- 代码分析与理解
- 设计方案输出
- 代码编写与修改
- 测试运行与验证
- Git 操作（分支管理、提交、PR 创建）

### 7.2 Agent 选型

- 当前支持：Claude Code
- 预留扩展能力，未来可接入其他 Agent

### 7.3 权限控制

Agent 执行操作需有权限管理：
- 常规操作（读写代码、运行测试）自动执行
- 危险操作（强制推送、删除分支等）需额外审批

**审批流程**：
```
Agent 需执行危险操作
    ↓
在 Issue Comment 发布审批请求：
  "需要执行 [操作名称]，原因：xxx，请仓库管理员回复 '同意' 执行"
    ↓
仓库管理员回复 "同意"
    ↓
Agent 执行操作
```

**超时策略**：
- 等待审批超过配置时间后，自动取消该操作
- 通知管理员操作已取消
- 流程暂停，等待管理员指令

**需审批的危险操作**：
- `git push --force` 强制推送
- `git branch -D` 删除分支
- 其他可配置的危险操作

## 8. 配置项

| 配置项 | 说明 | 默认值 |
|-------|------|--------|
| `maxConcurrency` | 最大并发 Issue 数 | 3 |
| `complexityThreshold` | 复杂度阈值（超过需人工确认） | 70 |
| `forceDesignConfirm` | 强制设计方案人工确认 | false |
| `taskRetryLimit` | Task 失败重试次数 | 3 |
| `agentType` | 使用的 Agent 类型 | `claude-code` |
| `approvalTimeout` | 审批等待超时时间 | 24小时 |
| `dangerousOperations` | 需审批的危险操作列表 | force-push, delete-branch |
| `clarityThreshold` | 需求清晰度阈值（低于需人工确认） | 60 |
| `clarityAutoProceedThreshold` | 清晰度自动进入设计阈值 | 80 |
| `clarityForceClarifyThreshold` | 清晰度强制继续澄清阈值 | 40 |
| `failure.retry.maxRetries` | 最大重试次数 | 3 |
| `failure.retry.initialBackoff` | 初始重试间隔 | 5s |
| `failure.retry.maxBackoff` | 最大重试间隔 | 60s |
| `failure.rateLimit.maxWaitDuration` | API 限流最大等待时间 | 30m |
| `failure.timeout.taskExecution` | 任务执行超时时间 | 30m |
| `failure.timeout.sessionMaxDuration` | 会话最大时长 | 72h |
| `failure.queue.maxLength` | 最大排队数量 | 10 |

## 9. 非功能需求

### 9.1 可观测性

- 每个 Issue 的处理日志
- 阶段耗时统计
- 错误告警通知

### 9.2 安全性

- GitHub Token 权限最小化
- 敏感操作需二次确认
- 代码变更审计日志

#### 9.2.1 审计日志

**记录范围**：
| 操作类型 | 说明 |
|---------|------|
| session_* | 会话生命周期事件（启动/暂停/恢复/终止） |
| stage_transition | 阶段转换 |
| agent_call | Agent 调用 |
| tool_use | 工具使用 |
| file_read/write | 文件操作 |
| bash_execute | Bash 命令执行 |
| git_operation | Git 操作 |
| pr_create | PR 创建 |
| approval_* | 审批相关（请求/决策） |

**查询功能**：
- 按时间范围查询
- 按仓库/会话过滤
- 按操作类型过滤
- 按操作者过滤
- 按结果过滤（成功/失败/拒绝）

**保留策略**：
- 默认保留 90 天
- 可配置保留天数
- 过期日志自动归档/清理

**查询 UI 功能**：
- 审计日志列表页面：支持多条件筛选和分页
- 审计日志详情页面：展示完整操作信息、参数、输出
- 支持按会话查看关联审计日志
- 支持按仓库查看审计日志

### 9.3 可扩展性

- Agent 抽象层支持接入其他 AI Agent
- 流程可配置（可跳过某些阶段）
